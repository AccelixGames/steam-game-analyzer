# 병렬 리뷰 분석 설계

## 목적

`/steam-insight` 스킬의 3-B 리뷰 분석 단계를 **서브에이전트 병렬 분석** 구조로 확장한다.
현재 LIMIT 10 × 3 = 30건 → 최대 ~1,300건(43배 커버리지 확대).

## 현재 상태

- 정성 분석(변경 전): `valid_reviews`에서 장문 긍정 10건 + 도움된 긍정 10건 + 부정 10건 = 최대 30건
- 정량 통계: `reviews` 테이블 직접 SQL (플레이타임별 긍정률, 언어별 분포)
- 데이터 규모: 102개 게임, 260,151건 유효 리뷰, 게임당 평균 2,550건

## 실행 환경

이 설계는 **코드 변경이 아닌 스킬 문서(SKILL.md) 변경**이다.
서브에이전트는 Claude Code의 `Agent` 도구로 디스패치한다:

- `Agent` 도구는 `model` 파라미터로 opus/sonnet 선택 가능
- `prompt` 파라미터에 게임 컨텍스트 + 리뷰 데이터 + 출력 템플릿을 모두 포함
- 여러 `Agent` 호출을 **단일 메시지에서 병렬 실행** 가능 (Claude Code 기본 기능)
- 각 서브에이전트의 결과는 메인 에이전트에게 텍스트로 직접 반환됨 (파일/DB 중간 저장 없음)

## 아키텍처

```
메인 Opus: 1단계 → 2단계 → 3-A 태그
    │
    ├── SQL: 상위 3개 언어 결정 (N개, 최대 3)
    ├── SQL: 각 파티션 리뷰 추출 (N×2개 × 200건 + TOP 100)
    ├── 게임 컨텍스트 요약 작성
    │
    ╞══ 병렬 디스패치 (최대 7개 동시) ═════════════╗
    │  Opus 서브에이전트 1개: TOP 100 심층 분석      │
    │  Sonnet 서브에이전트 ×(N×2): 언어별 패턴 분석  │
    │  (N = 실제 언어 수, 1~3. 총 에이전트 = 1+N×2) │
    ╞═════════════════════════════════════════════════╝
    │
    ├── 결과 통합 (아래 "결과 통합 규칙" 참조)
    ├── 3-B 정량 통계 (SQL — 기존 유지)
    ├── 4단계: 삼각 검증 (서브 insight 포함)
    ├── 5단계: 비즈니스
    ├── HTML 보고서 생성
    └── Footer에 에이전트 메트릭 기록
```

## 파티션 결정 (SQL)

### 상위 3개 언어 결정

```sql
SELECT language, count(*) as cnt
FROM valid_reviews WHERE appid = ?
GROUP BY language ORDER BY cnt DESC LIMIT 3;
```

### 파티션별 리뷰 추출

```sql
-- Sonnet 파티션: 언어별 긍/부정 TOP 200
SELECT recommendation_id, review_text, playtime_at_review,
       votes_up, weighted_vote_score, playtime_forever
FROM valid_reviews
WHERE appid = ? AND language = ? AND voted_up = ?
ORDER BY weighted_vote_score DESC
LIMIT 200;
```

### Opus용 TOP 100

```sql
SELECT recommendation_id, language, review_text, voted_up,
       playtime_at_review, votes_up, weighted_vote_score,
       playtime_forever
FROM valid_reviews
WHERE appid = ?
ORDER BY weighted_vote_score DESC
LIMIT 100;
```

## 서브에이전트 설계

### 공통 컨텍스트 (모든 서브에이전트에게 전달)

```markdown
## 게임 컨텍스트
- 게임명: {{name}}
- 장르: {{genres}}
- 태그 TOP 10: {{tags with vote counts}}
- 마케팅 한 줄 피치: {{short_description}}
- 핵심 메커니즘: {{2단계에서 메인이 파악한 코어루프 2~3문장}}
```

### Opus 서브에이전트 (TOP 100 심층 분석)

**모델:** opus
**입력:** 게임 컨텍스트 + TOP 100 리뷰 (weighted_vote_score 순)
**역할:** 개별 리뷰의 기획적 의미를 깊이 분석, 보고서 인용 후보 선정

**출력 템플릿:**

```markdown
### 기획 성공 포인트 (최대 5개)
각 포인트:
- **패턴명**: 한 줄 요약
- **증거 리뷰**: 원문 인용 (언어 · 플레이타임 · helpful수)
- **비영어면 번역 포함**
- **기획적 의미**: 왜 기획의 성공인가 (2~3문장)

### 기획 약점 (최대 5개)
(같은 형식)

### 흥미로운 발견 (최대 3개)
- 기획 성공/실패로 분류 어렵지만 주목할 패턴

### 보고서 인용 추천 리뷰 (최대 10개)
- recommendation_id, 원문(발췌), 인용 이유 1줄
```

### Sonnet 서브에이전트 (파티션 패턴 분석)

**모델:** sonnet
**입력:** 게임 컨텍스트 + 해당 파티션 리뷰 200건
**역할:** 200건에서 반복되는 패턴 발견 (개별 분석 아님)

**출력 템플릿:**

```markdown
### 핵심 패턴 (3~5개, 빈도순)
각 패턴:
- **패턴명**: 한 줄 요약
- **빈도**: 대략 몇 건에서 언급 (예: "약 40/200건")
- **대표 인용**: 패턴을 가장 잘 보여주는 리뷰 1건 (원문 발췌)
- **비영어면 번역 포함**
- **기획 시사점**: 이 패턴이 게임 설계에 대해 말해주는 것 (1~2문장)

### 이 언어/감정 고유의 특이점 (0~2개)
- 이 구역에서만 두드러지는 발견

### 요약 통계
- 가장 자주 언급된 게임 시스템/피처 TOP 3
- 감정 강도: 강한 표현 비율 체감
```

### Opus vs Sonnet 역할 분리

| | Opus (TOP 100) | Sonnet (파티션 200) |
|---|---|---|
| **목적** | 깊이 — 개별 리뷰의 기획적 의미 | 넓이 — 반복 패턴 발견 |
| **출력** | 인용 후보 + 기획 판정 | 패턴 + 빈도 + 시사점 |
| **메인 활용** | 보고서 인용, 삼각검증 증거 | 트렌드 파악, 기획축 분류 |

## 결과 통합 규칙

7개 서브에이전트의 결과가 텍스트로 반환된 후, 메인이 다음 규칙으로 통합한다.

### 패턴 병합

1. **동일 패턴 식별**: 여러 파티션에서 같은 게임 시스템/피처를 언급하면 하나로 병합
   - 예: EN-Positive와 KR-Positive 모두 "crafting system"을 핵심 패턴으로 → 하나의 "기획 성공" 포인트
2. **빈도 합산**: 병합된 패턴은 각 파티션의 빈도를 합산하여 전체 규모 추정
3. **대표 인용 선택**: 병합 시 가장 구체적이고 기획 시사점이 큰 인용 1~2개 채택

### 충돌 해결

- 긍정 파티션과 부정 파티션에서 **같은 시스템에 대해 상반된 평가**가 나오면 → "양면적 요소"로 기록 (삼각검증 4단계에서 핵심 인사이트)
- 언어별로 **상반된 반응**이면 → "문화권별 차이"로 기록

### 최종 보고서 반영량

- Opus 심층 분석에서: 기획 성공 최대 5개, 약점 최대 5개, 인용 추천 최대 10개
- Sonnet 패턴 분석에서: 파티션당 핵심 패턴 3~5개 → 병합 후 **보고서에 반영되는 고유 패턴은 최대 10개**
- 보고서 인용 리뷰 총 수: Opus 추천 10개 + Sonnet 대표 인용 중 선별 → **최종 15~20개**

### Opus 심층 분석과 Sonnet 패턴의 역할 분담

- **보고서 "기획 성공/약점" 섹션**: Opus 분석이 뼈대, Sonnet 패턴이 빈도 근거 보강
- **보고서 "리뷰 인용" 블록**: Opus 추천 리뷰 우선, Sonnet 대표 인용은 다른 언어/관점 보충 시 사용
- **삼각검증 테이블**: Opus가 발견한 기획 판정 + Sonnet의 빈도 데이터로 신뢰도 표기

## 에이전트 메트릭 추적

### 추적 항목 (에이전트별)

| 메트릭 | 측정 방법 |
|--------|----------|
| 입력 토큰 (추정) | 프롬프트 + 리뷰 텍스트 총 문자수 ÷ 3 |
| 출력 토큰 (추정) | 반환된 마크다운 문자수 ÷ 3 |
| 소모 시간 | 디스패치 전후 시간 차이 |
| 리뷰 건수 | 실제 전달된 리뷰 수 |
| 모델 | opus / sonnet |

> 정확한 API 토큰 카운트는 Agent 도구가 반환하지 않으므로, 문자 기반 추정 사용.
> 영어 ~4자/토큰, 한국어 ~2자/토큰, 혼합이므로 ÷3 일률 적용.

## 엣지 케이스

| 조건 | 동작 |
|------|------|
| `valid_reviews` ≤ 500건 | Fallback 모드 (아래 상세) |
| 언어 3개 미만 | 존재하는 언어만큼만 파티션 생성 (최소 2개: 1언어 × 긍/부정). 에이전트 수 = 1 + 언어수×2 |
| 파티션 200건 미만 | 있는 만큼 전달 |
| Opus TOP 100과 Sonnet 중복 | 허용 — 역할이 다름 (깊이 vs 패턴) |

### Fallback 모드 (≤ 500건)

서브에이전트 병렬 분석 **없이**, 메인 Opus가 직접 TOP 100건을 분석한다.

**동작:**
1. 메인이 `valid_reviews` TOP 100건을 SQL로 추출 (Opus용 TOP 100 쿼리와 동일)
2. 메인이 직접 리뷰를 읽고 기획 성공/약점/인용 후보를 판단
3. Sonnet 파티션 분석 없음 → 언어별/감정별 패턴 분석은 정량 SQL 통계로 대체
4. 보고서 Footer에 "Fallback 모드: 유효 리뷰 N건 (500건 이하), 리뷰 추가 수집 권장" 명시
5. 삼각검증은 동일하게 수행하되, 패턴 근거가 약해질 수 있음을 보고서에 명시

**사용자 안내:**
> "이 게임의 유효 리뷰는 N건입니다. 더 풍부한 분석을 위해 `/steam-crawl`로 리뷰를 추가 수집하세요."
